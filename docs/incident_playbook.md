## 1. Purpose

This playbook defines the standard operating procedures (SOPs) for handling security alerts generated by the Integrated SOC Lab environment. It covers:

- SIEM alert triage and classification
- Investigation procedures per attack type
- Escalation criteria and response actions
- Post-incident documentation

---

## 2. Alert Severity Tiers

| Tier | Severity | Response Time | Examples |
|------|----------|---------------|----------|
| **P1** | 🔴 CRITICAL | < 5 minutes | 10+ WAF blocks from single IP; active SQLi data exfiltration |
| **P2** | 🟠 HIGH | < 15 minutes | Suricata alert burst (>5 in 2 min); brute force campaign |
| **P3** | 🟡 MEDIUM | < 1 hour | Cross-layer correlation (same IP on WAF + NIDS) |
| **P4** | 🟢 LOW | Next shift | Scanner detection; single failed login |

## 4. Playbooks by Attack Type

### 4.1 SQL Injection (SQLi) — ModSecurity Detection

**Alert Name:** `CRITICAL: WAF Mass Block Alert`

**Investigation Steps:**
1. Open Splunk → `sourcetype="modsec:audit"` → filter by `src_ip`
2. Examine the `modsec_rule_id` field — is it a CRS rule (9xxxxx) or a custom rule (100xxx)?
3. Review the full request URI and POST body in the audit log
4. Check if the same IP triggered Suricata alerts (cross-layer)
5. Run: `| lookup threat_intel.csv src_ip OUTPUT ti_malicious`

**Response:**
- If `modsec_action = Intercepted`: WAF blocked the attack. **Log and monitor.**
- If data was returned (200 OK + suspicious body): **ESCALATE — potential data breach.**

---

### 4.2 Cross-Site Scripting (XSS) — ModSecurity Detection

**Alert Name:** `HIGH: XSS Payload Detected`

**Investigation Steps:**
1. Determine if XSS is Reflected or Stored
2. For stored XSS: Check feedback/comment endpoints for persisted payloads
3. Review `modsec_msg` for CRS rule details

**Response:**
- Reflected: WAF should block. Verify block rate.
- Stored: Higher severity — payload persists. Sanitize the database.

---

### 4.3 Brute Force — Rate Limiting Alert

**Alert Name:** `HIGH: Brute Force Threshold Exceeded`

**Investigation Steps:**
1. Query: `sourcetype="apache:access" uri="/login.php" | stats count by src_ip | where count > 10`
2. Cross-reference with failed login error logs
3. Check if any attempt succeeded (status=200 with "Welcome" in response)

**Response:**
- All failed: Add IP to blocklist. Check against threat intel.
- Any success: **P1 CRITICAL** — potential credential compromise. Force password reset.

---

### 4.4 Directory Traversal / LFI — Suricata + ModSecurity

**Alert Name:** `HIGH: Directory Traversal Attempt`

**Investigation Steps:**
1. Check Suricata `eve.json` for alert SID `1000005` or `1000006`
2. Review if `/etc/passwd` content appeared in any HTTP response
3. Cross-correlate with ModSecurity custom rule `100003`

**Response:**
- Blocked by WAF: Log and trend-track.
- File contents returned: **P1 CRITICAL** — sensitive file exposure.

---

### 4.5 Network-Level Alerts — Suricata NIDS

**Alert Name:** `MEDIUM: Cross-Layer Correlation Alert`

**Investigation Steps:**
1. Query: `sourcetype="suricata:eve" event_type="alert" | stats count by alert.signature, src_ip`
2. Compare with WAF data for the same `src_ip`
3. Review alert severity and category fields

**Response:**
- Single signature match: Monitor and trend.
- Multiple signatures + WAF correlation: **Upgrade to P2 HIGH.**

## 5. Post-Incident Actions

After every P1/P2 incident:

1. **Document** the incident in the alert store (`/alerts` endpoint)
2. **Capture** Splunk dashboard screenshots for evidence
3. **Update** detection rules if evasion was observed
4. **Conduct** a 15-minute debrief (simulated in the lab)

## 6. Key SPL Queries for Analysts

```spl
# Top attacking IPs across all sources
(sourcetype="modsec:audit" OR sourcetype="suricata:eve") | stats count by src_ip | sort -count | head 10

# Cross-layer correlation
(sourcetype="modsec:audit" OR sourcetype="suricata:eve") | stats dc(sourcetype) as layers count by src_ip | where layers > 1

# WAF block rate
sourcetype="modsec:audit" | eval action=if(match(_raw,"Intercepted"),"Blocked","Allowed") | stats count by action

# Failed login tracking
sourcetype="apache:error" "LOGIN_FAILED" | rex "user=(?P<username>\S+)" | stats count by username | sort -count

# Suricata alerts by severity
sourcetype="suricata:eve" event_type="alert" | stats count by alert.severity | sort -count
```

## 7. Escalation Matrix

| Condition | Action |
|---|---|
| Same IP flagged by WAF + NIDS | Escalate to P2 minimum |
| Successful SQLi (200 + data) | P1 — Immediate response |
| >50 alerts in 5 minutes from single IP | P1 — Possible automated attack |
| Threat intel match (known malicious IP) | Add context to ticket, escalate +1 tier |
| New attack pattern not covered by rules | Document and create custom rule |
